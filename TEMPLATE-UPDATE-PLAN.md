# Template Update Plan - GraphQL Mesh & Dependency Fixes

## Executive Summary

Based on the e2e2 dependency resolution work, we need to update our code generation templates to use correct, stable versions of GraphQL Mesh and related dependencies. The templates currently use non-existent or outdated versions.

## Current State Analysis

### Templates Identified
1. **`src/codegen/templates/bff/`** - Backend-for-Frontend with GraphQL Mesh
   - package.json.ejs
   - server.ts.ejs
   - tsconfig.json

2. **`src/codegen/templates/base-mfe/`** - Base MFE (Frontend + BFF combined)
   - package.json.ejs
   - (uses BFF templates for server-side)

3. **`src/codegen/UnifiedGenerator/unified-generator.ts`** - Generator logic
   - References template versions
   - May need updates for new API patterns

### Issues Found

#### 1. BFF Template (`src/codegen/templates/bff/package.json.ejs`)
**Current (BROKEN)**:
```json
"@graphql-mesh/cli": "latest",              // ❌ "latest" is unreliable
"@graphql-mesh/openapi": "latest",          // ❌ "latest" is unreliable
"@graphql-mesh/plugin-response-cache": "latest",
"@graphql-mesh/cache-localforage": "latest",
"@graphql-mesh/merger-stitching": "latest",
"@graphql-mesh/http": "latest",
"@graphql-mesh/merger-bare": "latest",
"helmet": "^7.1.0"                          // ⚠️ Too new, has breaking changes
```

**Problems**:
- Using "latest" can pull incompatible versions
- Missing peer dependencies (@graphql-tools/*)
- Missing tslib
- Helmet v7 has breaking changes (v8 is current but also has issues)

**Current (OUTDATED)**:
```json
"@graphql-mesh/cli": "^0.90.0",           // ❌ Too old (current: 0.100.21)
"@graphql-mesh/openapi": "^0.99.0",       // ❌ Too old (current: 0.109.26)

"@graphql-mesh/cache-localforage": "^0.99.0",
```

**Problems**:

#### 3. BFF Server Template (`src/codegen/templates/bff/server.ts.ejs`)
**Current (API MISMATCH)**:

**Problems**:

#### 4. TypeScript Config (`src/codegen/templates/bff/tsconfig.json`)
**Current (INCOMPATIBLE)**:
```json
{
  "module": "NodeNext",
  "moduleResolution": "NodeNext"
}
```

**Problems**:
- NodeNext resolution incompatible with Mesh v0.100 generated code
- Should use "commonjs" and "node" resolution

### Phase 3: Update BFF Server Template

**File**: `src/codegen/templates/bff/server.ts.ejs`

**Replace** the GraphQL handler section:

```typescript
// OLD (Remove):
import { getMesh } from '@graphql-mesh/runtime';
import { findAndParseConfig } from '@graphql-mesh/cli';
import { graphqlHTTP } from 'express-graphql';

let meshSchema: any;
(async () => {
  const meshConfig = await findAndParseConfig({ dir: __dirname });
  const mesh = await getMesh(meshConfig);
  meshSchema = mesh.schema;
})();

app.use('/graphql', async (req, res) => {
  if (!meshSchema) return res.status(503).json({ error: 'Mesh not ready' });
  return graphqlHTTP({ schema: meshSchema, ... })(req, res);
});

// NEW (Replace with):
import { createBuiltMeshHTTPHandler } from './.mesh';
import type { Request, Response, NextFunction } from 'express';

interface MeshContext {
  jwt?: string;
  requestId: string;
  userId?: string;
}

const meshHandler = createBuiltMeshHTTPHandler<MeshContext>();

// Add context middleware before GraphQL handler
app.use('/graphql', (req: Request, res: Response, next: NextFunction) => {
  // Attach context to request for Mesh
  (req as any).meshContext = {
    jwt: req.headers.authorization?.replace('Bearer ', ''),
    requestId: req.headers['x-request-id'] as string || crypto.randomUUID(),
    userId: extractUserIdFromToken(req.headers.authorization as string),
  };
  next();
}, meshHandler);
```

**Changes**:
- ✅ Use generated `.mesh` artifacts (created by `mesh build`)
- ✅ Use `createBuiltMeshHTTPHandler()` API (v0.100.x pattern)
- ✅ Remove express-graphql dependency
- ✅ Use middleware pattern for context injection
- ✅ Proper TypeScript types

## Open Questions

1. **Should we support both ESM and CommonJS?**
   - Current: Templates mix both (package.json has `"type": "module"` in BFF)
   - Recommendation: Stick with CommonJS for now (simpler, works with Mesh)

2. **Do we need mesh plugins (response-cache, localforage)?**
   - Current: BFF template includes them
   - Recommendation: Remove unless specifically required by DSL manifest

3. **Should version constants be in generator or separate config?**
   - Option A: In unified-generator.ts
   - Option B: Separate `src/codegen/dependency-versions.ts`
   - Recommendation: Option A (simpler, colocated)

4. **What about @types/helmet deprecation?**
   - Helmet v8 provides its own types
   - Recommendation: Remove @types/helmet from devDependencies
