# Daemon Integration & Refactor Plan

## Executive Summary

After comprehensive analysis of the `/Daemon` folder, I've identified a **critical architecture mismatch** between the existing component-based daemon and the MFE orchestration system. The daemon must be refactored from a **component router** to an **MFE orchestrator** to align with the project's goals.

## Current Daemon Architecture Analysis

### What Exists in `/Daemon` Folder

**Three-Service Component System:**
1. **Registry** (`component-system/registry/simple-registry.js`) - Node.js rule engine
2. **Daemon** (`component-system/daemon/rust/component-daemon/`) - Rust message router
3. **Renderer** (`component-system/renderer/frontend/`) - React UI

**Daemon Technology Stack:**
- **Runtime**: Tokio (async Rust)
- **HTTP Server**: Warp
- **GraphQL**: async-graphql + async-graphql-warp
- **WebSocket**: tungstenite + tokio-tungstenite
- **Protocol**: graphql-transport-ws (modern GraphQL over WebSocket)
- **State**: DashMap (concurrent HashMap)
- **Messaging**: tokio::sync::broadcast (pub/sub)

### Current Daemon Responsibilities

**Component Model** (NOT MFE):
```rust
struct Component {
    id: String,              // UUID
    type: String,            // "CARD", "NOTIFICATION", "FORM"
    data: serde_json::Value, // Arbitrary JSON
    createdAt: DateTime,
}

struct Action {
    id: String,
    componentId: String,
    actionType: String,      // "CLICK", "SUBMIT"
    data: serde_json::Value,
    timestamp: DateTime,
}
```

**Message Flow:**
```
Registry → Daemon (WS subscription) → Renderer (WS subscription)
  └─ Components generated by rules, broadcast to renderers

Renderer → Daemon (GraphQL mutation) → Registry (HTTP POST)
  └─ User actions normalized and forwarded
```

**Key Operations:**
- Routes messages between registry and renderers
- Maintains per-component state + action history (in-memory)
- Normalizes action types (`BUTTON_CLICK` → `CLICK`)
- Broadcasts `STATE_SNAPSHOT` after mutations
- Auto-reconnects to registry with exponential backoff
- Provides GraphQL Playground for debugging

### What the Daemon Does NOT Do

❌ Load MFEs or Module Federation containers
❌ Fetch or transform DSL manifests
❌ Perform server-side rendering
❌ Serve static assets or bundles
❌ Manage MFE dependencies or capabilities
❌ Handle multi-MFE workflows
❌ Validate orchestration states
❌ Track MFE health or performance

## Critical Architecture Mismatch

| Aspect | Current Daemon (Component) | Required System (MFE) |
|--------|---------------------------|----------------------|
| **Entity** | UI Components (JSON data) | Micro-Frontends (Module Federation) |
| **Data Model** | `Component { type, data }` | `MFEDescriptor { manifestUrl, remoteEntryUrl }` |
| **Loading** | N/A (components are data) | Fetch DSL, load remote entry, instantiate |
| **State** | Per-component action history | Per-MFE load state + capabilities |
| **Orchestration** | Rule-based component generation | Multi-MFE workflow coordination |
| **Dependencies** | None | Capability-based dependency graph |
| **Registry** | Component generator (rules) | MFE metadata store (manifests, health) |
| **Protocol** | GraphQL subscription (components) | WebSocket (MFE events) + HTTP (manifests) |
| **Rendering** | Browser renders JSON as UI | Browser loads federated modules |
| **Lifecycle** | Create → Update → Destroy | Load → Initialize → Ready → Unload |

**Conclusion:** The daemon must be **refactored from component routing to MFE orchestration**.

---

## Refactor Strategy: Component Daemon → MFE Orchestrator Daemon

### Design Principle

**Separation of Concerns:**
- **Daemon (Rust)**: MFE registry, health monitoring, metadata serving, workflow coordination
- **Control Package (TypeScript/Browser)**: MFE loading, rendering, execution, capability validation

**The daemon should be the "source of truth" for MFE metadata, the browser should be the "execution engine".**

### Phase 6: Align WebSocket Protocol with DaemonClient (2-3 hours)

**Replace GraphQL-Transport-WS with raw WebSocket:**

```rust
async fn handle_websocket(ws: WebSocket, daemon: Arc<MFERegistryDaemon>) {
    let (mut ws_tx, mut ws_rx) = ws.split();
    let mut daemon_rx = daemon.subscribe();

    // Forward daemon messages to client
    tokio::spawn(async move {
        while let Ok(msg) = daemon_rx.recv().await {
            let json = serde_json::to_string(&msg).unwrap();
            ws_tx.send(Message::text(json)).await.ok();
        }
    });

    // Handle client messages
    while let Some(result) = ws_rx.next().await {
        match result {
            Ok(msg) => {
                if let Ok(daemon_msg) = serde_json::from_str::<DaemonMessage>(msg.to_str()?) {
                    handle_client_message(daemon_msg, &daemon).await;
                }
            }
            Err(_) => break,
        }
    }
}
```

**Message Handling:**
- `RegistryQuery` → Send current MFE list
- `MFELoadRequest` → Fetch and return manifest
- `Ping` → Respond with `Pong`
- Auto-broadcast on registry changes

```typescript
private async createMFEInstance(descriptor: MFEDescriptor): Promise<RemoteMFE> {
    try {
        let manifestData: unknown;

        // Prefer daemon registry if available
        if (this.registryClient) {
            logger.info('Fetching manifest from daemon registry', { mfeId: descriptor.id });
            manifestData = await this.registryClient.getManifest(descriptor.id);
        } else {
            // Fallback: direct fetch
            logger.info('Fetching manifest directly', { url: descriptor.manifestUrl });
            const response = await fetch(descriptor.manifestUrl);
            if (!response.ok) {
                throw new Error(`Failed to fetch manifest: ${response.statusText}`);
            }
            manifestData = await response.json();
        }

        // Parse manifest using DSL package
        const { parseManifestFile } = await import('@seans-mfe-tool/dsl');
        const manifest = parseManifestFile(manifestData);

        // Create RemoteMFE instance
        const { RemoteMFE } = await import('@seans-mfe-tool/runtime');
        const mfe = new RemoteMFE(manifest, {
            telemetry: this.telemetry,
            platformHandlers: {},
        });

        logger.info('Created RemoteMFE instance', {
            mfeId: descriptor.id,
            name: manifest.name,
        });

        return mfe;
    } catch (error) {
        logger.error('Failed to create MFE instance', {
            mfeId: descriptor.id,
            error: (error as Error).message,
        });
        throw error;
    }
}
```

## Key Architectural Decisions

### 1. Why Remove the Registry Service?
- MFEs are external services, not generated by rules
- No need for component generation logic
- Simpler: Daemon = Registry (reduces service count from 3 to 1)
- MFEs self-register on startup via HTTP POST

### 2. Why Keep Rust for the Daemon?
- Excellent async performance (Tokio)
- Strong typing with serde serialization
- Concurrent state management (DashMap)
- Low resource usage
- Matches existing daemon infrastructure

### 3. Why Separate Daemon from Browser Runtime?
- **Daemon**: Centralized metadata, multi-tenant, long-lived
- **Browser**: Per-user session, loads/executes MFEs, ephemeral
- Clear separation: Daemon = registry/metadata, Browser = execution

### 4. Why Replace GraphQL-Transport-WS with Raw WebSocket?
- TypeScript `DaemonClient` uses raw WebSocket (not GraphQL client)
- Simpler protocol, less overhead
- GraphQL still available via HTTP for queries/mutations
- WebSocket for real-time pub/sub only

### 5. Why Keep GraphQL?
- Great for exploratory queries (GraphQL Playground)
- Subscriptions for real-time updates (registry changes, health)
- Optional - clients can use HTTP REST instead
