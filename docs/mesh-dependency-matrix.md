# GraphQL Mesh Dependency Matrix

**Document Status:** ✅ Current  
**Last Updated:** April 2026  
**Requirement:** REQ-MESH-DEPS-005  
**ADR Reference:** ADR-062 (GraphQL Mesh v0.100.x with Production Plugins & Transforms)

---

## Executive Summary

This document records the tested and validated dependency compatibility matrix for the GraphQL Mesh
BFF layer generated by `seans-mfe-tool`. The matrix was established through dependency analysis
completed in December 2025 (ADR-062) and remains the canonical source of truth for template
package generation.

### Migration Decision

The original issue proposed migrating to:
```json
{
  "@graphql-mesh/runtime": "^0.100.21",
  "@graphql-mesh/compose-cli": "^0.5.x",
  "@omnigraph/openapi": "^0.4.x"
}
```

After evaluation (see [bff-refactor-analysis.md](./bff-refactor-analysis.md)), the team chose
**not** to migrate to `compose-cli`/`omnigraph` and instead locked to the stable **v0.100.x CLI**
series (ADR-062). The rationale:

- `@graphql-mesh/compose-cli` is a v1-era package with a separate, incompatible API surface
- `@omnigraph/openapi` was still in early development with incomplete documentation
- `@graphql-mesh/cli@^0.100.21` + `@graphql-mesh/serve-runtime@^1.2.4` covers the needed use case
- The v0.100.x series is actively maintained and stable

## Validated Version Matrix

All versions below are tested for `npm install` compatibility (no ERESOLVE errors) and
TypeScript compilation success. They are centralized in
`src/codegen/UnifiedGenerator/unified-generator.ts → DEPENDENCY_VERSIONS`.

### Core GraphQL Mesh

| Package                      | Version    | Notes                                          |
| ---------------------------- | ---------- | ---------------------------------------------- |
| `@graphql-mesh/cli`          | `^0.100.21`| Stable CLI — runs `mesh build`, `mesh dev`     |
| `@graphql-mesh/openapi`      | `^0.109.26`| OpenAPI handler (actively maintained)          |
| `@graphql-mesh/serve-runtime`| `^1.2.4`   | HTTP handler — `createBuiltMeshHTTPHandler()`  |

> **Note:** `@graphql-mesh/serve-runtime` follows a separate version scheme from the CLI.
> v1.2.4 is compatible with `@graphql-mesh/cli@^0.100.21`.

### GraphQL Tools (Peer Dependencies)

These must be explicitly declared — Mesh handlers rely on them but do not pin their own versions.

| Package                    | Version    | Notes                                  |
| -------------------------- | ---------- | -------------------------------------- |
| `@graphql-tools/delegate`  | `^10.2.4`  | Required by Mesh for type merging      |
| `@graphql-tools/utils`     | `^9.2.1`   | Mesh internal utility layer            |
| `@graphql-tools/wrap`      | `^10.0.5`  | Required by schema transforms          |
| `graphql`                  | `^16.8.1`  | Peer dependency for all Mesh packages  |

> **Warning:** Do NOT mix `graphql@15.x` with the above — peer constraint violations will occur.

## TypeScript Compatibility

**Critical configuration** — the Mesh runtime requires CommonJS module resolution:

```json
{
  "compilerOptions": {
    "module": "commonjs",
    "moduleResolution": "node",
    "skipLibCheck": true
  }
}
```

> **Do NOT use** `"module": "NodeNext"` or `"moduleResolution": "NodeNext"` with Mesh v0.100.x.
> This causes import resolution failures for Mesh's `.mesh/` generated artifacts.

## Incompatible Combinations (Do NOT Use)

| Combination | Problem |
| ----------- | ------- |
| `@graphql-mesh/cli@0.100.x` + `@omnigraph/openapi` | Different handler interface, incompatible `.meshrc.yaml` format |
| `@graphql-mesh/compose-cli` + `@graphql-mesh/cli@0.100.x` | Two competing CLI tools, conflicting `mesh` binary |
| `graphql@15.x` + any `@graphql-mesh/*` package | Peer dependency violations, TypeScript type errors |
| `react@^18.x` (caret) in MFE context | Can load React 18.1 vs 18.3 across federation boundary |
| `moduleResolution: "NodeNext"` + Mesh | Generated `.mesh/` artifacts fail to resolve |
| `@graphql-tools/utils@^10.x` + `@graphql-mesh/cli@0.100.x` | Utils API change breaks Mesh internals (use `^9.2.1`) |

## Migration Status vs. Original Issue

The original REQ-MESH-DEPS-005 proposed a four-phase migration:

| Phase | Description | Status | Notes |
| ----- | ----------- | ------ | ----- |
| Phase 1 | Baseline testing (current v0 setup) | ✅ Complete | Done via DEPENDENCY-PLAN.md & ADR-062 |
| Phase 2 | Add `@graphql-mesh/runtime` alongside CLI | ✅ Complete | `serve-runtime` added (different package, same purpose) |
| Phase 3 | Swap `@graphql-mesh/openapi` → `@omnigraph/openapi` | 🚫 Not taken | ADR-062 locked to `^0.109.26` — `@omnigraph` not stable |
| Phase 4 | Remove `@graphql-mesh/cli` | 🚫 Not taken | CLI remains the entry point for `mesh build` / `mesh dev` |

The proposed migration target (`@graphql-mesh/runtime` + `@graphql-mesh/compose-cli` +
`@omnigraph/openapi`) was a v1-era API that was still in flux at the time of evaluation.
ADR-062 codified the decision to stay on the stable v0.100.x release train.

## Automated Validation

Run the dependency compatibility test script:

```bash
node scripts/test-mesh-dependencies.js
```

This script validates:
1. Version range consistency between all Mesh packages
2. Known incompatible combinations are absent from templates
3. Template `package.json.ejs` uses versioned deps (not `"latest"`)
4. `DEPENDENCY_VERSIONS` in `unified-generator.ts` matches template references

## Source of Truth

The version constants are maintained in:

```
src/codegen/UnifiedGenerator/unified-generator.ts → DEPENDENCY_VERSIONS
```

Template files consume these via EJS interpolation:
```
packages/bff-plugin/templates/package.json.ejs
```

Do **not** hardcode versions in template `.ejs` files — always reference `DEPENDENCY_VERSIONS`.
